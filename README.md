

# The Ghost in the Machine — Mechanistic AI Text Detection

Author: **P Sam Sahil**

---

## Overview

This project explores a **mechanistic interpretability approach** to detecting AI-generated text.

Instead of relying on surface-level statistical features (like lexical richness or perplexity), the system analyzes the **internal representations of a large language model** to determine whether a piece of text was written by a human or generated by an AI.

The key idea is simple but powerful:

> Instead of classifying the text itself, we classify the **model's internal conceptual representation of the text**.

By extracting activations from **Gemma-2-2B**, transforming them into **interpretable Sparse Autoencoder (SAE) features**, and training an **XGBoost classifier**, we obtain a highly accurate AI-text detector while maintaining interpretability.

---

# Project Pipeline

The full pipeline works as follows:

```
Text
 ↓
Tokenization (Gemma-2-2B tokenizer)
 ↓
Embedding Layer
 ↓
Transformer Forward Pass
 ↓
Layer-20 Residual Stream Extraction
 ↓
Sparse Autoencoder (SAE) Feature Projection
 ↓
Max Pooling Across Tokens
 ↓
Document Concept Vector
 ↓
XGBoost Classifier
 ↓
Human vs AI Prediction
```

This architecture allows us to analyze **what conceptual features inside the model fire differently for human and AI writing**.

# Key Idea

Typical AI detectors rely on:

• Perplexity
• Stylometric features
• Embeddings

These methods often detect **surface artifacts**.

This project instead analyzes the **internal neuron-level feature space** of a transformer model.

Sparse Autoencoders convert dense hidden states into **interpretable concept features**, where each dimension corresponds to a semantic pattern.

Examples of patterns detected by SAE features include:

• formal explanatory tone
• statistical references
• structured explanation style
• instructional language patterns
• discourse markers and transitions

Because these features represent **actual interpretable directions in model space**, we can inspect them using **Neuronpedia**.

# Why Gemma-2-2B?

Gemma was chosen because:

1. **Pretrained SAEs exist for it**
2. Its internal features are mapped on **Neuronpedia**
3. It is large enough to capture meaningful semantic structure
4. It is computationally feasible to run locally

Other models considered:

| Model       | Reason Not Used                         |
| ----------- | --------------------------------------- |
| GPT-2 Small | Too small, features unreliable          |
| Llama 3 70B | Too large for limited compute           |
| Gemma 2     | Best balance of size + interpretability |

---

# Why Layer 20?

Transformers process information hierarchically.

Early layers:
• capture token patterns
• local syntax

Middle layers:
• encode semantic meaning
• conceptual structure

Final layers:
• focus on next-token prediction

Layer **20** sits in the **semantic processing region**, making it a strong candidate for extracting meaningful representations.

Future work could explore aggregating across **layers 15-25** for improved robustness.

---

# Sparse Autoencoders (SAE)

Sparse Autoencoders convert dense transformer activations into **interpretable sparse feature vectors**.

Instead of thousands of entangled neuron activations, SAEs produce a representation where:

• each dimension corresponds to a **specific concept**
• most values remain **zero**
• only meaningful features activate

This produces a **concept activation map** for the document.

Example conceptual features detected:

• explanatory tone
• enumerations and lists
• statistical references
• formal discourse structure

---

# Document Fingerprint

Each token produces its own SAE feature vector.

To obtain a **document-level representation**, we apply:

```
Max Pooling across tokens
```

This keeps the strongest activation of each feature anywhere in the text.

The result is a **fixed-size sparse vector representing the entire document**.

---

# Classifier

The classifier used is **XGBoost**.

Reasons for choosing XGBoost:

• works extremely well with sparse features
• handles nonlinear interactions between concepts
• robust to overfitting
• interpretable feature importance

The classifier learns decision rules like:

```
IF feature_8573 is high AND feature_6107 is low
→ likely AI-generated
```

# Interpretability

The most important property of this system is **interpretability**.

Unlike standard detectors:

• each feature corresponds to a **real neuron direction**
• every feature can be inspected on **Neuronpedia**
• we can trace **why the model predicts AI text**

This turns AI detection from a **black box classification problem** into a **mechanistic analysis problem**.

---

# Future Work: Adversarial Attacks

One interesting implication of this approach is that the classifier can potentially be bypassed.

Since we know which SAE features trigger the detector, an attacker could modify generation to suppress them.

Possible methods:

### Feature Clamping

Force specific SAE features to zero during generation.

### Gradient Steering

Apply gradient-based methods to steer generation away from detector-triggering features.

Related work:

• *Using Mechanistic Interpretability to Craft Adversarial Attacks against LLMs* (2025)
• *Steering Language Model Refusal with Sparse Autoencoders* (2024)

This suggests a future research direction:

> Mechanistic AI detection vs mechanistic adversarial attacks.
